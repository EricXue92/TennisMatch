# i18n Phase 6–11 — 设计文档

**日期**：2026-04-24
**状态**：Draft → 待用户 review
**前置文档**：`docs/superpowers/specs/2026-04-24-app-localization-design.md`（Phase 1 原始设计）

## 1. 目标

完成 App 语言切换功能的收尾：**移除 English 选项**，并把剩余未本地化的 View 全部纳入 **繁體中文 ↔ 简体中文** 切换范围。

Phase 1–5 已完成：
- 基础设施（`LocaleManager`、`L10n`、`Localizable.xcstrings`、`.environment(\.locale)` 注入、Settings 语言 picker）
- 已本地化：TabBar、SettingsView、HomeView、MyMatchesView、ProfileView、MessagesView、ChatDetailView

本文档覆盖 Phase 6–11 — 6 个后续阶段，每个阶段一个 branch、一个 PR，按依赖顺序递进。

## 2. 范围决策（来自 brainstorming）

### 2.1 移除 English —— 为什么

用户决定 App 只保留 **繁體中文（source）+ 简体中文**。原因：目标用户是华人业余网球爱好者，English 维护成本高、覆盖面低。

### 2.2 为什么要做完整覆盖

半本地化的 App 比纯繁中更糟：用户切到简中后，若 Login、CreateMatch、Tournament 仍显示繁中，切换功能看起来像坏了。繁↔简 本质是字符变体映射（單→单、網→网、約→约），单 key 成本远低于翻译英文，因此把剩余 View 跑完一遍是合理的。

### 2.3 Login/Onboarding 优先级

系统语言为 zh-Hans 的新用户，默认走「跟隨系統」分支，第一眼就是 Login。此时他们还进不了 Settings 改语言，所以 Login/SignUp/Onboarding 的简中覆盖必须优先完成。

## 3. 架构（沿用 Phase 1）

本阶段**不**改基础架构。沿用：

- `LocaleManager`（@Observable，UserDefaults 持久化）
- `.environment(\.locale, localeManager.currentLocale)` 根节点注入
- `Localizable.xcstrings`（source = zh-Hant）
- SwiftUI 视图中 `Text("繁中字面量")` 自动本地化
- 非 SwiftUI 场景（toast、alert message 赋值、model 动态字符串）走 `L10n.string(_:)`

唯一基础改动：`LocaleManager.AppLanguage` 移除 `.en` case（见 Phase 6）。

## 4. Phase 拆分

每个 Phase = 一个 git branch + 一个 PR，命名沿用 `feat/i18n-phase{N}-{topic}`。

### Phase 6 — LoginView + RegisterView + 移除 English（picker/enum 层）

**优先级**：最高（新用户首屏）

**范围**：
- `LocaleManager.AppLanguage` 删除 `.en` case
- `LocaleManager.currentLocale` 对应 switch 分支删除
- SettingsView 语言 picker 删除 `Text("English").tag(.en)` 行
- 扫描 `LoginView.swift` 及其关联 subview —— 把所有 `Text("...")`、`navigationTitle(...)`、`Button("...")`、`alert(...)`、`TextField` placeholder 的繁中字面量注册到 xcstrings，补 zh-Hans 翻译
- 扫描 `RegisterView.swift` 同理（项目中实际文件名为 `RegisterView.swift`，即注册流程入口）
- 所有 `toastMessage = "..."` 形式的赋值改为 `toastMessage = L10n.string("...")`
- 所有 helper 函数的 `title: String` 参数，如果传的是本地化字面量，改成 `LocalizedStringKey`（参考 Phase 1 的 `settingsRow`/`tappableRow` 改法）

**不做**：xcstrings 里 `en` 的历史翻译条目不动（按 Phase 11 的 soft deprecation 策略永久保留为休眠数据）。

**验收**：Simulator 切到「简体中文」后，冷启动进入 Login → 所有文案为简中；切回「繁體中文」→ 所有文案为繁中；切「跟隨系統」+ 系统 zh-Hans → 简中。

### Phase 7 — CreateMatchView + MatchAssistantView + MatchDetailView

**优先级**：高（核心日常操作）

**范围**：
- `CreateMatchView.swift`（发布约球）、`MatchAssistantView.swift`（约球助手/一键约球入口）、`MatchDetailView.swift`（约球详情）及关联 form 子 view（日期/时间/人数/水平/球场 picker）
- 所有 form 字段 label、section header、placeholder、validation error toast、成功提示
- 日期/时间选择器的 label（`DatePicker("...")` 等 `LocalizedStringKey` 接口）
- 注意：date picker 的 *格式化输出* 由 `.locale` 环境驱动，通常无需代码改动；但如有 `.formatted()` 自定义 format string，需改为走 `.locale` 环境

**不在本 Phase**：`CreateTournamentView` 归属 Phase 9；`NTRPGuideView` / `NTRPRangeSlider` 归属 Phase 8。

### Phase 8 — NTRPGuideView + NTRPRangeSlider + Onboarding（如有）

**优先级**：高（首次使用流程 + NTRP 教育内容）

**范围**：
- `NTRPGuideView.swift` —— NTRP 分级说明文字（每个级别的描述段落）
- `NTRPRangeSlider.swift` —— slider 交互 label 及其标签文字
- Onboarding 引导页文案（若项目中存在独立 Onboarding 流程；Phase 实施时先 grep 确认入口，如未找到则本 Phase 仅覆盖 NTRP 两个文件）
- 「我的水平」自评流程的所有交互文案（通常分散在 ProfileView / RegisterView 中 —— 前者已在 Phase 4 做过，后者在 Phase 6 做，本 Phase 只处理 NTRP 专属内容）

**注意**：NTRP 描述段落可能是长文本。确认每段是一个 xcstrings key，而非多段拼接 —— 拼接会导致 key 爆炸和翻译断句错误。

### Phase 9 — Tournament 系列

**优先级**：中（低频使用）

**范围**：
- `TournamentView.swift`（赛事列表/详情）、`CreateTournamentView.swift`（创建赛事）及关联子 view
- 赛事状态 badge（「报名中」「已开赛」「已结束」等）
- 赛事相关的 confirm alert、toast、报名/取消流程文案

### Phase 10 — Models 内动态字符串

**优先级**：中

**范围**：
- `NotificationStore` 通知文案模板（例：「{user} 邀请你打 {matchType}」）
- `CalendarService` 日历事件标题/描述
- 评论/评级相关模板（如「{user} 给你的评分是 {rating}」）
- 任何返回 `String` 供 View 展示的 Model 层逻辑

**技术要点**：
- Model 层无 SwiftUI env 访问，必须走 `L10n.string(_:)`
- 带参 key 用 `String.LocalizationValue` 插值：`L10n.string("\(user) 邀請你打\(matchType)")` → xcstrings 自动识别为 `"%@ 邀請你打%@"`
- 检查现有 xcstrings 已有的 `"%@ 邀請你打%@"` 等 key（Xcode 已自动扫描），补 zh-Hans 翻译即可

### Phase 11 — 日期/数字格式 + English 软下线（保留翻译数据）

**优先级**：低（polish + chore）

**设计决策（soft deprecation）**：保留 xcstrings 中已有的 `"en"` 翻译数据以便未来重新启用 English 时无需从零翻译；仅从 project 层面和 UI 层面下线。

**范围**：
- 全仓库扫描硬编码的 `DateFormatter` / `NumberFormatter` / `.formatted(.dateTime.…)` 调用
  - 若 formatter 未指定 locale，依赖系统 locale —— 这意味着**用户切到简中但系统是繁中时，日期仍是繁中格式**
  - 统一方案：formatter 的 `locale` 属性显式设为 `LocaleManager.shared.currentLocale`；SwiftUI 场景下 `Text(date, format: ...)` 会自动用 env locale，无需改动
- **English 软下线**：
  - 在 Xcode Project → Info → Localizations 移除 English（手动 UI 操作，会把 `en` 从 `knownRegions` 中剔除，阻止后续 build 再往 xcstrings 自动追加 `en` 列）
  - 确认 `TennisMatch.xcodeproj/project.pbxproj` 的 `knownRegions` 不再包含 `en`
  - **不**删除 `Localizable.xcstrings` 中已有的 `"en"` stringUnit 条目 —— 作为休眠资产保留
  - Phase 6–10 过程中 Xcode 已自动为新 key 生成的 `en` 空条目可以原状保留（不影响运行）
- 全量 Simulator 走查：繁↔简 切换所有已本地化的页面，确认无遗漏

**验收**：
- Settings 语言 picker 只剩 3 个选项（跟随系统/繁體中文/简体中文）
- `project.pbxproj` 的 `knownRegions` 不含 `en`
- 冷启动后无论系统 locale 为何（包括 en-US），App 均 fallback 到繁中源语言
- xcstrings 文件里历史 `"en"` 翻译仍在（这是预期行为，非遗漏）

### Phase 11.5 — 未来重启 English 支持的 checklist（保留以备查）

如未来决定重新启用 English，按顺序执行：
1. `LocaleManager.swift`：`AppLanguage` enum 重新加回 `case en`；`currentLocale` switch 加回 `case .en: return Locale(identifier: "en")`
2. `SettingsView.swift` 通用 section 的 Picker 重新加回 `Text("English").tag(LocaleManager.AppLanguage.en)`
3. Xcode Project → Info → Localizations → `+` → English（pbxproj 自动更新 `knownRegions`）
4. 打开 `Localizable.xcstrings`，Build 一次 —— Xcode 会把 Phase 6–10 新增的 key 自动补 `en` 空条目（state: new）
5. 在 Xcode String Catalog 编辑器筛选 state = new 的 `en` 条目，逐个补翻译；Phase 1–5 的条目因本次软下线保留，无需重做
6. Simulator 走查 en 模式

## 5. 通用约定（所有 Phase 遵循）

### 5.1 繁↔简 翻译原则

- 繁→简 是**字符变体映射**而非翻译，保留原句结构
- 常见映射：單→单、網→网、約→约、場→场、個→个、點→点、發→发、關→关、認→认、證→证、實→实、歲→岁、確→确、設→设、這→这、麼→么、後→后、裡→里、號→号、當→当、經→经、選→选、業→业、類→类、雙→双、聯→联、響→响
- 不做简体化之外的改写（「影片」繁、「视频」简这种二区习惯差异**不做**；一律字符映射）
- 疑难字查 Xcode String Catalog 的翻译建议或现有翻译一致性

### 5.2 何时用 `Text(...)` vs `L10n.string(...)`

| 场景 | 用法 |
|---|---|
| SwiftUI View body 内 `Text`、`Label`、`Button`、`navigationTitle`、`alert title` 等接受 `LocalizedStringKey` | `Text("繁中字面量")` — 自动本地化 |
| 赋值给 `String` 变量（toast、state、非本地化接口） | `L10n.string("繁中字面量")` |
| Helper 函数 `title: String` 参数 | 改签名为 `title: LocalizedStringKey` |
| Model 层（无 env 访问）返回给 View 的字符串 | `L10n.string(_:)` |

### 5.3 Xcstrings workflow

1. 在代码里写繁中字面量
2. Build 一次 —— Xcode 自动扫描并把 key 加到 xcstrings（state: new）
3. 在 Xcode 打开 xcstrings 编辑器，切到 zh-Hans 列，为新 key 填简中翻译（state: translated）
4. Phase 6–10 期间：**忽略 English 列**。即使 Xcode 自动为新 key 生成空 en 条目也不必填 —— 它们会随 Phase 11 的软下线一起保留为休眠数据，未来若启用 English 再补。

### 5.4 Commit 约定

沿用 Phase 1–5：
- `feat(i18n): Phase N — <topic>` 用于 View 层本地化
- `refactor(i18n): route <file> toasts through L10n.string` 用于 `L10n.string` 迁移
- `chore(i18n): xcstrings auto-scan additions from Xcode build` 用于单独收纳 Xcode 自动扫描结果（如有）

## 6. 出入标准（Definition of Done，全部 Phase）

- [ ] Settings 语言 picker 只剩 3 项：跟隨系統 / 繁體中文 / 简体中文
- [ ] 用户切到「简体中文」后，冷启动 + 热切换，全 App 每个可达页面均显示简中
- [ ] 同上，切到「繁體中文」全显繁中
- [ ] 「跟隨系統」在系统为 zh-Hans 时显示简中，zh-Hant 时显示繁中，其他语言 fallback 到繁中（source）
- [ ] Notification 推送文案（Phase 10）也跟随 LocaleManager 切换
- [ ] 日期/时间格式在两种语言下视觉一致（繁中「4月24日」/ 简中「4月24日」—— 两者通常相同，但需验证）
- [ ] `project.pbxproj` 的 `knownRegions` 不含 `en`
- [ ] `xcstrings` 中 `"en"` 翻译条目**保留**作为休眠资产（soft deprecation，未来可重启）
- [ ] 无 build warning / error
- [ ] 每个 Phase 的 PR 已 merge 到 main

## 7. 不在范围内（Out of Scope）

- ❌ **本 spec 内**重启 English —— 软下线后可随时重启，流程见 Phase 11.5 checklist
- ❌ 新增其他语言（日语、韩语等）—— 如未来要加，基础设施已支持，按 Phase 2+ 模式追加即可
- ❌ 改变 source 语言（保持 zh-Hant 不变）
- ❌ RTL 语言支持
- ❌ 地域性数字/货币格式化（NTRP 分数、场次数字在中文圈内表现一致，无需区分）
- ❌ App Store metadata 本地化（App 名称、描述、截图 —— 属于 App Store Connect 侧工作）
- ❌ 单元测试（项目无 XCTest target）

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 某 Phase 翻译遗漏，用户切简中仍看到繁中 | 每 Phase 走查清单（所有 Text/Label/Button/alert/toast/placeholder/section header） |
| xcstrings 在 Xcode 里编辑时自动重排/格式化，产生大 diff noise | Commit 前 `git diff --stat` 确认；必要时单独 commit 成 `chore(i18n): xcstrings normalization` |
| Model 层 `L10n.string` 依赖 `LocaleManager.shared` 单例 —— 切换瞬间的时序问题 | `@Observable` 变更会触发 View 重 render；Model 层字符串在被 View 读取时才解析，时序天然正确 |
| 代码不再引用 English，但 xcstrings 里仍有 `"en"` 条目 | **这是有意的 soft deprecation**（Phase 11 决策）；休眠数据保留以便未来重启 English，不算遗漏 |
| NTRP 长段落翻译人力成本 | 由用户（开发者本人）逐段填写；spec 预估每个段落 100–200 字，共 ~7 级，总量可接受 |

## 9. 后续 Phase 的写 plan 流程

每个 Phase 开始时：
1. 从 main 切新 branch：`git checkout -b feat/i18n-phase{N}-{topic}`
2. 调用 `superpowers:writing-plans` skill，基于本 spec 的对应 section 生成实施 plan
3. 实施 → build → simulator 走查 → commit → push → PR → merge

本 spec 不替代 plan；每个 Phase 仍需单独 plan 拆步骤。
