# 约球流程压力测试 & 逻辑找茬报告

> 视角:资深 iOS 测试 + 网球老炮
> 日期:2026-04-26
> 范围:发布 → 报名 → 确认 → 聊天 完整生命周期

---

## 一、最致命的 3 个逻辑漏洞

### 🔴 #1 「确认环节」没有兜底机制 → 用户体验崩塌点

流程是「报名 → 发起人接受」。但发起人**不点**会怎样?

- 报名者被卡在 `pending` 状态,既不能确定行程,也不敢报别的局。
- 结合首页 `已報名未滿員` 的 pending/confirmed 区分,这个状态长尾问题会越来越严重。

**必须做**:

- 距开球 T-12h 仍未审核 → **自动接受** or **自动释放**(二选一,建议前者,因为业余局多数发起人就是「先到先得」心态)。
- 距开球 T-2h 局未满 → 推送给发起人「再不决定就开不成了」。
- pending 报名者要能**主动撤回**而不被惩罚(因为不是他的锅)。

### 🔴 #2 时间冲突 + 取消信用分 没有形成闭环

已经踩过「取消約球必須清 externalSlots」的坑——说明**时间冲突检测**已经在做了。但缺一块:

- **临开球取消的惩罚**和**冲突检测**是两件事,目前都只在客户端层面。
- 一个用户在 A 局 pending、又去 B 局 confirmed → 是按哪个算占用?如果 pending 不占,他可以海投然后挑;如果 pending 占,A 局发起人不接受就锁死他的时间。

**建议规则**:

- `pending` **软占用**(UI 提示冲突,可强制提交)
- `confirmed` **硬占用**(直接禁报)
- 距开球 < 4h 取消 confirmed → 信用分 -10;爽约不到场 → -30
- 信用分 < 60 → 高级局/认证发起人的局禁止报名

### 🔴 #3 并发抢最后一个名额 → 客户端先校验必翻车

SwiftUI + 本地 mock 的现状下,「人数已满」大概率是**渲染时判断**。真接后端时,两个 3.5 同时点「报名」,客户端都看到「还剩 1 名」,都通过本地校验提交 → 后端如果没做乐观锁/原子计数,**会双双成功**或**互相覆盖**。

**必须**:

- 名额扣减在服务端用 `UPDATE ... WHERE remaining > 0` 原子操作。
- 客户端收到 `409 SlotTaken` 后,UI 要从「报名成功」**优雅回退**到「已被抢走」,而不是闪一下成功再变红。
- 这个 case 的视觉反馈现在大概率没设计——可以提前画一版。

## 二、「发布 → 报名 → 聊天」全生命周期 Checklist

### 📝 发布阶段

- [ ] 必填:开球时间、结束时间、球场(venue+具体court)、人数、NTRP区间、费用、AA/请客
- [ ] 选填:球性(单打/双打/混双)、用球品牌、是否提供球
- [ ] 时间合法性:不能发布过去时间;开球时间距现在 < 1h 时二次确认
- [ ] 球场冲突:同一 court 同一时段不允许重复发布(需要球场级锁)
- [ ] 草稿保存:中途切走不丢失
- [ ] 发布后可编辑范围:**仅备注/费用**可改;时间/地点改动 = 视为重新发布,所有报名人需重新确认

### 🎯 报名阶段

- [ ] NTRP 不在区间 → 弹窗劝退而非硬拦(写明「发起人要求 4.0+,你是 3.0,确定报名?」)
- [ ] 时间冲突检测(覆盖已 confirmed + pending + tournament + 外部 booking)
- [ ] 信用分门槛检测
- [ ] 报名后可取消:T-24h 无惩罚,T-4h 扣分,T-1h 视为爽约
- [ ] 报名状态明确分层:`pending / confirmed / waitlist / rejected / cancelled`
- [ ] 候补队列:满员后允许排队,有人取消自动递补 + 推送

### 💬 确认 + 聊天阶段

- [ ] 被接受 → **强 push** + App 内红点 + 自动加入群聊
- [ ] 群聊入口至少 3 处:push 跳转、约球详情页、「我的约球」列表
- [ ] 群聊**不要**报名成功就建——应在**发起人接受后**建,避免拒绝后还能聊
- [ ] 球局取消 → 群聊保留 24h 供退款/善后,之后归档
- [ ] 赛后 2h 触发互评卡片(在原群聊里以系统消息出现,不另开页面)
- [ ] 评价可匿名 / NTRP 校准结果只对自己可见

### 🚨 必须 Push 的节点

| 事件             | 接收人           | 紧急度                    |
| ---------------- | ---------------- | ------------------------- |
| 报名被接受       | 报名者           | 高                        |
| 球局被发起人取消 | 全体已 confirmed | 致命(走静默推送+短信兜底) |
| 候补递补成功     | 候补者           | 高                        |
| T-2h 球局未满    | 发起人           | 中                        |
| 新群聊消息       | 群成员           | 低(可聚合)                |
| 赛后互评提醒     | 全员             | 低                        |

## 三、提升粘性的 2 个差异化建议

### 🎾 #1 「球友熟练度档案」—— 比 NTRP 更细的真实人设

NTRP 是个数字,匹配靠它远远不够。打球的人最在意的是**对方什么风格**:暴力上旋?切削老大爷?网前型?

- 互评维度细分:正手 / 反手 / 发球 / 网前 / 移动 / 战术 / **球品**
- 累计 5 次以上评价后,资料页生成**雷达图 + 「TA 的标签」**(比如「反手切削大师」「捡球积极」「容易跑神」)
- 这个东西**只有你的 App 有**——其他约球软件都还停在数字 NTRP。
- 副作用:解决「自评虚高」(校准提示),用客观评价反推。

### 🎾 #2 「固定搭子」+ 复盘日记

业余球员最大的痛点其实不是「找一次球」,而是**「找到能长期一起打的人」**。

- 一键把同场打过 3+ 次的人加为「固定搭子」
- 搭子之间发起的局自动获得高优先级匹配 / 免审核 / 默认 confirmed
- 每场球后自动生成一张「战绩卡片」(对手、比分、场地、时长),可一键分享朋友圈
- **复盘日记**:打完手动记录「今天反手不稳」,App 用这些数据帮你追踪进步轨迹
- 这一步把 App 从「工具」变成「网球生活记录本」,留存率会上一个台阶。
